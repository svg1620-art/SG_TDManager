# SG_T&D_manager — Техническое задание

**Продукт:** таск-трекер и биллинг часов для методологов на аутсорсе (ServiceGuru)
**Версия ТЗ:** 1.0
**Язык интерфейса:** русский
**Назначение документа:** источник истины для постадийной сборки в Claude Code

---

## 1. Цель и контекст

Клиенты, подключившие платформу ServiceGuru, покупают услугу «Методолог на аутсорсе» на 3 / 6 / 12 месяцев. Методолог настраивает платформу клиента: контент (курсы, тесты, обложки, видео), меню, фуд-пары, логика уведомлений, опросы, добавление сотрудников, аттестации, отчёты.

**Проблема:** вся коммуникация идёт в чатах, клиенты накидывают задачи без учёта лимита по часам, методолог перерабатывает сверх оплаченного бюджета.

**Решение:** SG_T&D_manager — платформа, где:
- клиент в личном кабинете ставит задачи, видит остаток лимита по часам, статус задач и общается по каждой задаче;
- методолог ведёт единое поле задач по всем своим клиентам, логирует реальные часы, закрывает задачи, видит дашборды нагрузки;
- администратор видит сводку по всем методологам, распределяет клиентов, добавляет часы, контролирует переработки и фрод.

Ключевой механизм против перегруза: **предварительная оценка часов методологом + обязательное подтверждение клиентом** до взятия задачи в работу.

## 2. Роли и права доступа

| Роль | Права |
|------|-------|
| **Администратор** (ты) | Видит всё: всех методологов, всех клиентов, все задачи. Переназначает клиентов между методологами. Выставляет и корректирует месячные лимиты, добавляет часы. Может закрывать задачу без подтверждения клиента. Видит фрод-монитор (часы по методологам за день). Управляет учётками методологов и клиентов. |
| **Методолог** | Видит своих закреплённых клиентов и их задачи. Двигает статусы задач, ставит оценку, логирует время, закрывает задачи. Заводит клиентов и их сотрудников, выставляет лимиты, добавляет часы (по факту оплаты по счёту — вручную). Видит свои дашборды нагрузки и распределения по клиентам. |
| **Клиент (сотрудник)** | Принадлежит организации-клиенту. Создаёт задачи, комментирует, подтверждает оценку, принимает/возвращает выполненную задачу. Видит месячный лимит и расход, оценку и факт по каждой задаче, дашборд и месячный отчёт. |

**Правила закрепления:**
- Клиент закреплён строго за одним методологом, но может быть передан другому (переназначение админом).
- При переназначении все задачи клиента переходят к новому методологу; исторические записи времени сохраняют исполнителя (для отчётности).
- Все сотрудники одного клиента видят **все** задачи своей организации; различается только фамилия постановщика (создателя) в карточке.
- Лимит часов — **общий на организацию-клиента**, не делится по сотрудникам.

## 3. Жизненный цикл задачи

Основной пайплайн двигает **методолог**. У клиента два действия-«шлюза»: подтверждение оценки и приёмка результата.

```
[Клиент создаёт]
      │
   Новая ──(методолог ставит оценку часов)──> Ожидает подтверждения
                                                     │
                          (клиент подтверждает оценку)│
                                                     ▼
                                              Готова к работе ──(методолог берёт)──> В работе
                                                                                        │
                                             ┌──────────────────────────────────────────┤
                                             ▼                                          ▼
                                        На уточнении ──(разъяснено)──> В работе      Выполнена
                                                                                        │
                                              ┌─────────────────────────────────────────┤
                              (клиент принимает / админ закрывает)         (клиент возвращает)
                                              ▼                                          ▼
                                          Принята (финал)                            В работе
```

**Статусы (хранимые):**
- `new` — Новая (создана клиентом)
- `estimate_pending` — Ожидает подтверждения (методолог поставил оценку часов, ждём клиента)
- `approved` — Готова к работе (клиент подтвердил оценку, в очереди)
- `in_progress` — В работе
- `clarification` — На уточнении (методолог заблокирован, нужен ответ клиента в комментариях)
- `done` — Выполнена (методолог закрыл, ждём приёмки клиентом)
- `accepted` — Принята (финал)
- `rejected` — Отклонена (финал; часы не списывает)

**Действия клиента (не свободное управление статусами):**
- Подтвердить оценку (`estimate_pending` → `approved`).
- Принять выполненную (`done` → `accepted`) или Вернуть на доработку (`done` → `in_progress`, с обязательным комментарием-причиной).

**Действия администратора:**
- Может перевести `done` → `accepted` без подтверждения клиента.
- Может отклонить/переоткрыть любую задачу.

**Правила оценки часов:**
- Оценку ставит методолог на этапе `new` → `estimate_pending`.
- Методолог может изменить оценку при закрытии задачи и исправить её у уже закрытой задачи.
- При подтверждении оценки клиенту показывается предупреждение, если задача выведет его за месячный лимит (но подтвердить всё равно можно — см. раздел 4).

## 4. Учёт часов и лимиты

**Модель лимита:**
- Лимит выставляется **на месяц, индивидуально по каждому клиенту** (клиенты разного размера и важности).
- Неизрасходованные часы **сгорают** — переноса на следующий месяц нет.
- При создании профиля клиента задаётся `default_monthly_hours`. На 1-е число каждого месяца APScheduler автоматически создаёт запись месячного бюджета из значения по умолчанию (если методолог не задал иное вручную).

**Расчёт расхода:**
- `Расход за месяц` = сумма часов всех записей времени (`time_entries`) с `work_date` внутри этого месяца, по всем задачам клиента.
- `Остаток` = `Эффективный лимит` − `Расход`. Может уходить в минус.
- `Эффективный лимит` = базовый лимит месяца + сумма ручных добавлений часов за этот месяц.

**Поведение при исчерпании лимита:**
- Клиент **не блокируется** в создании задач — задачи копятся в очереди.
- Если методолог берёт такую задачу в работу и логирует время, **расход уходит в минус** (осознанное решение методолога).
- В интерфейсах методолога и админа минусовой остаток подсвечивается.

**Добавление часов (расширение тарифа):**
- Ручное действие методолога или админа (оплата — по счёту, вне платформы, ЮKassa на этом этапе не интегрируем).
- Каждое добавление логируется в аудит: клиент, месяц, +N часов, причина/комментарий, кто добавил, когда.

## 5. Кабинет клиента

**Дашборд (побольше графики):**
- Гейдж/прогресс-бар: месячный лимит vs расход на текущий момент (с числами: лимит, потрачено, остаток).
- Донат по статусам задач (сколько новых, в работе, выполнено, принято и т.д.).
- Донат распределения потраченных часов по типам работ.
- Счётчики: всего задач за месяц, активных, выполнено.

**Список задач:**
- Карточки/строки со статусом, приоритетом, типом работ, постановщиком (фамилия сотрудника-создателя).
- В каждой задаче видны **оценка** и **факт** часов.
- Фильтры по статусу/типу/приоритету.

**Карточка задачи (клиент):**
- Название, описание, приоритет, тип работ, статус, оценка/факт.
- Кнопки-шлюзы по статусу: «Подтвердить оценку», «Принять», «Вернуть на доработку» (с комментарием).
- Лента комментариев (текстовые, с указанием автора и времени).

**Создание задачи (клиент):** название, краткое описание, приоритет (Низкий / Средний / Высокий / Срочный), тип работ (см. раздел 8). Вложений на этом этапе **нет**.

**Месячный отчёт:** просмотр текстового отчёта за месяц (генерит нейросеть, см. раздел 10) с возможностью выгрузки.

## 6. Рабочее место методолога

**Единое поле задач:**
- Все задачи по всем закреплённым клиентам в одном месте.
- Вид-доска по колонкам-статусам (перемещение по кнопкам действий, без drag-and-drop) + вид-список.
- Фильтры: клиент, статус, приоритет, тип работ.

**Карточка задачи (методолог):**
- Просмотр всех полей + постановщик и клиент.
- Управление статусами по пайплайну (раздел 3).
- Поле оценки часов (ставит/меняет).
- **Трекинг времени — набор записей:** добавить запись (часы, дата, короткая заметка). Записи суммируются в факт по задаче. Живого таймера нет.
- Кнопка «Закрыть задачу» → перевод в `Выполнена`, с возможностью скорректировать оценку.
- Информация по лимиту клиента и текущему расходу прямо в карточке.
- Лента комментариев.

**Дашборды методолога:**
- Общий баланс работ (сколько часов законтрактовано/лимитов у его клиентов vs фактически потрачено в текущем месяце).
- Распределение трудозатрат по клиентам (бар/донат).
- Подсветка клиентов в минусе и близких к лимиту.

**Управление клиентами (методолог):**
- Создание профиля клиента: название, `default_monthly_hours`, срок контракта (3/6/12 мес, даты начала/конца), `telegram_chat_id` группы.
- Привязка нескольких учётных записей сотрудников к клиенту.
- Выставление/корректировка месячного лимита, добавление часов (с логом).

> Заведение учёток клиентов: методолог создаёт логин/пароль и передаёт их клиенту со ссылкой в Telegram **вне платформы** (задача методолога, не системы). Самостоятельной регистрации клиентов нет.

## 7. Кабинет администратора

- Обзор всех методологов: количество клиентов, суммарная нагрузка, задачи в работе.
- Переназначение клиентов между методологами (задачи переходят вместе с клиентом).
- Выставление/корректировка месячных лимитов и добавление часов любому клиенту.
- Закрытие/приёмка задач без подтверждения клиента.
- Глобальные дашборды: распределение нагрузки по методологам и клиентам, клиенты в минусе.

**Фрод-монитор (контроль переработок методолога):**
- Таблица «часы по методологам за день»: суммарное залогированное время каждого методолога по всем задачам за текущий момент/за день.
- Подсветка, если дневная сумма методолога превышает норму рабочего времени (порог настраивается, дефолт 8 ч/день) — сигнал о возможном фроде/приписках.

**Типы работ:** `content` Контент · `setup` Настройка · `communication` Коммуникация · `platform` Работа с платформой · `attestation` Аттестации · `reports` Отчёты · `other` Прочее.

## 9. Уведомления

**Внутренний колокольчик (in-app):**
- Клиенту: новая оценка (нужно подтвердить), смена статуса, новый комментарий, задача возвращена/принята.
- Методологу: новая задача от клиента, подтверждение/возврат клиентом, новый комментарий, задача на уточнении разъяснена.
- Хранятся в таблице `notifications`, помечаются прочитанными.

**Telegram «пульс» (push-only, без команд):**
- Раз в день APScheduler отправляет дайджест в группу клиента (по `telegram_chat_id`) через Telegram Bot API.
- Содержимое: что создано/закрыто за день, потрачено часов, остаток лимита.
- 5-го числа в пульс добавляется **краткая версия месячного отчёта**.
- Один бот-токен на всю систему (в env), рассылка по `chat_id` каждого клиента.

## 10. AI-отчёт (ежемесячный)

- **Триггер:** APScheduler, **5-е число** каждого месяца — генерация отчёта за прошедший месяц по каждому активному клиенту. Плюс кнопка ручной перегенерации для методолога/админа.
- **Движок:** Anthropic Claude API, модель **Sonnet**.
- **Вход в модель:** перечень выполненных за месяц задач, часы по типам работ, сравнение с прошлым месяцем, использование лимита.
- **Выход:** деловой текстовый отчёт клиенту: перечень проделанной работы, разбивка часов по типам, динамика к прошлому месяцу, краткие рекомендации. Тон — деловой отчёт заказчику.
- **Хранение:** таблица `monthly_reports` (текст). Доступ на просмотр и выгрузку в кабинете клиента и у методолога. Формат выгрузки — текст (.txt/.md).
- Краткая выжимка отчёта дублируется в Telegram-пульс 5-го числа.

## 11. Технический стек и инфраструктура

- **Бэкенд:** Flask + Jinja2 + HTMX (минимум клиентского JS).
- **БД:** PostgreSQL (SQLAlchemy ORM).
- **Графики:** Chart.js.
- **Планировщик:** APScheduler (создание месячных бюджетов 1-го числа, пульс-дайджест ежедневно, AI-отчёты 5-го числа).
- **AI:** Anthropic Claude API (Sonnet).
- **Telegram:** прямые вызовы Bot API для push-рассылки (aiogram не требуется — интерактива нет).
- **Хостинг:** Railway, **регион EU обязателен** (доступ к Anthropic API).
- **Репозиторий:** GitHub, редактирование через web UI (полная замена файлов по стадиям).

**Constraints (зашить в CLAUDE.md):**
- Railway **EU region** — обязательно.
- `DATABASE_URL` — только внутренняя reference-переменная Railway, не публичный URL руками.
- Файловая система Railway **эфемерна** — на этом этапе персистентного файлового хранилища не нужно (вложений нет, отчёты хранятся в БД как текст).
- Env-переменные добавлять по одной через «+ New Variable», без кавычек, без multiline-вставки, без кириллицы в значениях.
- `httpx` пинить под версию SDK `anthropic`, чтобы избежать конфликтов совместимости.
- Если Claude вернёт структурированный ответ в markdown-fence — извлекать парсером, а не наивным `json.loads()` (для текстовых отчётов не критично, но правило действует).

**Env-переменные:**
`DATABASE_URL`, `SECRET_KEY`, `ANTHROPIC_API_KEY`, `TELEGRAM_BOT_TOKEN`, `ADMIN_LOGIN`, `ADMIN_PASSWORD` (сид первого админа), `FRAUD_DAILY_HOURS_THRESHOLD` (дефолт 8).

**Фирменный стиль:**
- Тёмная база `#0A0A0D`, синий `#1467F5`, циан `#00BFDC`.
- Шрифты: Manrope (дисплей), JetBrains Mono (числовые данные).

## 12. Модель данных

**users**
`id, role(admin|methodologist|client), full_name, login, password_hash, email, client_id(FK→client_orgs, только для роли client), is_active, created_at`

**client_orgs**
`id, name, methodologist_id(FK→users), telegram_chat_id, default_monthly_hours(numeric), contract_months(3|6|12), contract_start(date), contract_end(date), is_active, notes, created_at`

**monthly_budgets**
`id, client_id(FK), year(int), month(int), base_limit_hours(numeric), created_by(FK→users), created_at` — уникальность (client_id, year, month). Эффективный лимит = base_limit_hours + Σ budget_adjustments за месяц.

**budget_adjustments** (аудит добавления часов)
`id, client_id(FK), year(int), month(int), delta_hours(numeric), reason(text), created_by(FK→users), created_at`

**tasks**
`id, client_id(FK), created_by(FK→users, сотрудник клиента), title, description, priority(low|medium|high|urgent), work_type(content|setup|communication|platform|attestation|reports|other), status(new|estimate_pending|approved|in_progress|clarification|done|accepted|rejected), estimated_hours(numeric), actual_hours(numeric, кэш = Σ time_entries), created_at, updated_at, closed_at, accepted_at, return_reason`

**time_entries** (набор записей времени)
`id, task_id(FK), methodologist_id(FK→users), hours(numeric), work_date(date), note(text), created_at`

**comments**
`id, task_id(FK), author_id(FK→users), body(text), created_at`

**notifications**
`id, user_id(FK), type, task_id(FK, nullable), body, is_read(bool), created_at`

**monthly_reports**
`id, client_id(FK), year(int), month(int), body_text(text), generated_at, source(auto|manual)` — уникальность (client_id, year, month).

## 13. Постадийный план сборки (для Claude Code)

Каждая стадия = один сеанс = полная замена файлов, деплой на Railway EU через GitHub.

- **Стадия 1 — Каркас.** Flask + SQLAlchemy + PostgreSQL, все модели, миграции/`create_all`, аутентификация (логин/пароль, роли, сессии), сид первого админа из env, базовый layout с фирменным стилем (тёмная тема, Manrope/JetBrains Mono), деплой на Railway EU.
- **Стадия 2 — Клиенты и бюджеты.** CRUD организаций-клиентов (методолог/админ), привязка сотрудников, месячные лимиты, `default_monthly_hours`, автосоздание бюджета на 1-е число (APScheduler), добавление часов с аудитом.
- **Стадия 3 — Задачи.** Создание клиентом, полный жизненный цикл и статусы, оценка часов + подтверждение клиентом, приёмка/возврат, комментарии, колокольчик (базовые in-app уведомления).
- **Стадия 4 — Трекинг времени.** `time_entries` (набор записей), пересчёт факта и кэша `actual_hours`, расчёт расхода/остатка по месяцу, уход в минус, отражение лимита в карточке.
- **Стадия 5 — Дашборды.** Кабинет клиента с графикой (Chart.js: гейдж лимита, донаты статусов и типов работ), единое поле задач методолога (доска + список + фильтры), баланс и распределение по клиентам.
- **Стадия 6 — Админка и фрод-монитор.** Обзор методологов, переназначение клиентов, глобальные дашборды, таблица часов по методологам за день с подсветкой превышения порога.
- **Стадия 7 — Telegram-пульс.** Ежедневный push-дайджест в группы клиентов через Bot API (APScheduler), `telegram_chat_id` в профиле клиента.
- **Стадия 8 — AI-отчёт.** Claude API (Sonnet), генерация 5-го числа по каждому клиенту, ручная перегенерация, хранение и выгрузка текста, краткая версия в пульс.
