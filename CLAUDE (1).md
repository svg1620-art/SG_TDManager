# CLAUDE.md — SG_T&D_manager

Память проекта для сеансов Claude Code. Читать перед каждой стадией. Все правила ниже — обязательны.

---

## Что это

Таск-трекер и биллинг часов для методологов ServiceGuru, работающих на аутсорсе. Клиенты ставят задачи в личном кабинете, видят остаток лимита по часам, статус и общаются по каждой задаче. Методолог ведёт единое поле задач, логирует реальные часы, закрывает задачи. Админ видит сводку по методологам, распределяет клиентов, добавляет часы, ловит переработки/фрод.

Полное ТЗ — в `SG_T&D_manager_spec.md`. Этот файл — рабочая выжимка правил.

---

## Стек (не менять без причины)

- **Backend:** Flask + Jinja2 + HTMX (минимум клиентского JS)
- **ORM/БД:** Flask-SQLAlchemy + PostgreSQL
- **Auth:** Flask-Login, пароли через `werkzeug.security`
- **Планировщик:** APScheduler
- **Графики:** Chart.js (через CDN)
- **AI:** Anthropic Claude API, модель Sonnet
- **Telegram:** прямые вызовы Bot API через `requests` (push-only, без aiogram)
- **Сервер:** gunicorn
- **Хостинг:** Railway, регион **EU** (обязательно)
- **Репозиторий:** GitHub, правки через web UI

---

## Жёсткие правила инфраструктуры

1. **Railway EU region — обязательно.** Нужен для доступа к Anthropic API.
2. **`DATABASE_URL`** — только внутренняя reference-переменная Railway. Никогда не вставлять публичный URL руками.
3. **Файловая система Railway эфемерна.** Никакого локального хранения файлов. В v1 вложений нет, отчёты храним в БД как текст. Volume не заводим.
4. **Env-переменные** добавлять по одной через «+ New Variable». Без кавычек, без multiline-вставки, без кириллицы в значениях.
5. **`httpx==0.27.2`** пиновать вместе с SDK `anthropic`, чтобы не ловить конфликт совместимости.
6. Если Claude вернёт структуру в markdown-fence (```json```) — извлекать парсером, не наивным `json.loads()`. Для текстовых отчётов не критично, но правило действует.
7. Схему БД создаём через `db.create_all()` при старте (миграций нет — правки через GitHub web UI, полная замена файлов).

## Роли

- **admin** — видит всё, переназначает клиентов, выставляет лимиты/добавляет часы, закрывает задачи без клиента, фрод-монитор.
- **methodologist** — свои клиенты и их задачи, двигает статусы, оценка, логирование времени, закрытие, заводит клиентов/сотрудников/лимиты.
- **client** — сотрудник организации-клиента: создаёт задачи, комментирует, подтверждает оценку, принимает/возвращает выполненную.

Клиент закреплён за одним методологом, может быть передан другому (задачи переходят вместе с ним). Сотрудники одного клиента видят все задачи организации; различается фамилия постановщика. Лимит — общий на организацию.

## Ключевые бизнес-правила

**Учёт часов:**
- Лимит — на месяц, индивидуально по клиенту. Неизрасходованные часы **сгорают** (переноса нет).
- Расход месяца = Σ `time_entries.hours`, где `work_date` попадает в месяц (атрибуция по дате записи, не по дате закрытия).
- Эффективный лимит = базовый лимит месяца + Σ ручных добавлений часов за месяц.
- При исчерпании лимита задачи не блокируются — копятся в очереди; если методолог берёт в работу, расход уходит в минус (подсвечивать).
- Добавление часов — ручное действие методолога/админа, с аудитом (оплата по счёту вне платформы; ЮKassa не интегрируем).
- 1-го числа APScheduler автосоздаёт месячный бюджет из `default_monthly_hours`, если не задан вручную.

**Жизненный цикл задачи** (пайплайн двигает методолог; у клиента два шлюза):
`new → estimate_pending → approved → in_progress → (clarification ↔ in_progress) → done → accepted | rejected`
- Методолог ставит оценку: `new → estimate_pending`.
- Клиент подтверждает оценку: `estimate_pending → approved`.
- Методолог берёт: `approved → in_progress`; закрывает: `in_progress → done` (может скорректировать оценку).
- Клиент принимает `done → accepted` или возвращает `done → in_progress` (с обязательным комментарием).
- Админ может закрыть `done → accepted` без клиента.
- Оценку можно менять при закрытии и у уже закрытой задачи.

**Трекинг времени:** набор записей (часы, дата, заметка), суммируются в `actual_hours`. Живого таймера нет.

**Фрод-монитор:** суммарные залогированные часы каждого методолога за день; подсветка при превышении `FRAUD_DAILY_HOURS_THRESHOLD`.

**Справочники:**
- Приоритеты: low / medium / high / urgent.
- Типы работ: content / setup / communication / platform / attestation / reports / other.

## Уведомления

- **Колокольчик (in-app):** новая задача, новая оценка, смена статуса, новый комментарий, возврат/приёмка. Таблица `notifications`.
- **Telegram-пульс (push-only):** ежедневный дайджест в группу клиента по `telegram_chat_id` через Bot API. 5-го числа — краткая версия месячного отчёта.

## AI-отчёт

- APScheduler, 5-е число: генерация за прошлый месяц по каждому активному клиенту (+ ручная перегенерация).
- Claude Sonnet. Вход: выполненные задачи, часы по типам, сравнение с прошлым месяцем, использование лимита. Выход: деловой текстовый отчёт с рекомендациями.
- Хранение: `monthly_reports` (текст), выгрузка текстом. Краткая выжимка — в пульс.
